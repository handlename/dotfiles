各見出しの括弧書きは、それぞれの重要度である。
重要度の定義は以下の通り。

- MUST: 必ず守られなければならない
- SHOULD: 望ましいが、ユーザーの指示やプロジェクトの規約と矛盾する場合はそちらを優先せよ
- MAY: 心構え

# (MUST) git または jujutsu の選択

`jj st` コマンドで jujutsu が使用できるリポジトリかどうかを確認せよ。
jujutsu が使用できる場合は優先して使用せよ。

# (MUST) 適切な説明用言語の選択

ドキュメントやソースコードへのコメントなど、人間が読むことが想定されるテキストには、適切な言語を選択せよ。
既存のドキュメント(README.md など)が存在する場合は、それらで使用されている言語を採用せよ。
セッション内でユーザーが使用する言語とは、必ずしも一致しないことを留意せよ。

# (MUST) agent が作成したドキュメントの無断コミット禁止

明確な指示がある場合を除いて、agent が作成したドキュメント(design doc, plan など)のコミットを禁ずる。
それらのドキュメントはIssue・Pull Requestの説明文やコメント、コミットコメントに記載すれば十分である。

# (MUST) GPG 署名のないコミットの禁止

すべてのコミットには GPG 署名を行うこと。
1Password の応答がないことにより署名に失敗する場合は、ユーザーによる対応を求めること。
この場合、エージェント判断によるリトライや、タスクの失敗判定を行ってはならない。

# (MUST) 許可のない外部リソース操作の禁止

ワーキングディレクトリ外への変更を伴う操作は、ユーザーの許可なく実行してはならない。
例: git push、http POST/PUT/PATCH リクエスト

# (MUST) sub-agent に調査・レビューを依頼する際はその結果をファイルに書き出させる

sub-agent には構造化された調査結果をファイルに書き出させ、そのパスを要求すること。

# (MUST) 冗長なコメントの禁止

単純な処理(単発の関数呼び出しなど)に対する冗長なコメントは禁ずる。

# (MUST) どのドキュメントをどこに書くのか、の指針

以下に基づいて適切な内容、記載場所を選択する事。

- コードには How
- テストコードには What
- コミットログには Why
- コードコメントには Why not

# (SHOULD) Always-Valid Domain Model の実践

ドメイン層を実装する際は、Always-Valid Domain Model を意識せよ。

# (SHOULD) Functional Core, Imperative Shell

操作のコア部分は、Functional Core として実装せよ。
永続層の操作は、Imperative Shell として実装せよ。

# (SHOULD) 純粋関数の使用

積極的に純粋関数を使用せよ。
副作用がない純粋関数は、テストしやすく、動作の把握も容易である。
ただし、既存関数の純粋関数化は、指示があるまで行ってはならない。

# (SHOULD) immutable な型の利用

積極的に immutable な型を使用せよ。
immutable な型は、変更ができないため、不変性を保ち、並列処理や並列化が容易である。
ただし、指示にない既存の型の immutable 化は禁ずる。

# (SHOULD) コードの意図を補足するコメントの記述

コードのみからではその意図が伝わりにくい処理には、その意図を説明するコメントを記述せよ。

# (SHOULD) 動作の正しさの主張前に実データで検証する

動作が正しいと推論するだけでなく、実データを用いたシミュレーションで裏付けを取ること。
特にデータ処理やダッシュボードの変更では、既存のデータファイルを使った再現検証が有効である。

# (SHOULD) git コマンドを実行する際は --no-pager フラグを付与する

pager の使用によりコマンドの実行が終了せず、その後の処理に進めなくなってしまう。
git コマンドには --no-pager フラグを付与し、pager を使わずその出力を確認すること。

# (SHOULD) 可能な限りシンプルなアプローチを使用する

明示的に求められない限り、新しいインターフェース、型、抽象化を追加しないこと。
このタスクの直接的なスコープ外のコードをリファクタリングまたは変更しないこと。
複数のアプローチがある場合は、複雑性が最も低いものを選択すること。

# (MAY) GitHub CLI を用いた GitHub 上リソースへのアクセス

`github-resource-access` skill に従え。
